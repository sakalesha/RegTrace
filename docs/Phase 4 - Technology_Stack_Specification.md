## PART 2 — FRONTEND STACK

- **React.js**: Enables a component-based architecture for the internal compliance dashboards. It handles state management (via Context/Redux/Zustand) and UI reactivity.
- **TypeScript**: Ensures type safety across the application, significantly reducing runtime errors when mapping backend JSON responses to frontend models.
- **Vite**: Replaces Create React App/Webpack to provide a significantly faster developer experience with hot-module replacement (HMR) and optimized esbuilds.
- **Tailwind CSS**: Allows developers to style components directly using utility classes without maintaining sprawling custom CSS files, ensuring a consistent design language.

**Integration with Backend:**
The frontend will communicate with the FastAPI backend via Axios or Fetch API using standard REST protocols, secured by JWTs passed in the `Authorization` header.

**Recommended Project Structure:**
`/src/components` (reusable UI), `/src/pages` (route views), `/src/hooks` (custom React hooks), `/src/services` (API HTTP clients), `/src/types` (TypeScript interfaces).

---

## PART 3 — BACKEND STACK

- **Python**: Selected universally for the backend to eliminate context-switching between the AI/ML pipeline and the standard CRUD APIs.
- **FastAPI**: Selected for its exceptional performance, native `async/await` support, and seamless integration with Pydantic for data validation.

**Responsibilities & Features:**
- **Dependency Injection**: FastAPI natively supports DI, which will be heavily used for injecting DB sessions, Redis clients, and AI agents into route handlers.
- **Validation**: Incoming requests and outgoing responses are strictly validated using Pydantic models (DTOs).
- **OpenAPI Support**: FastAPI automatically generates interactive Swagger documentation, fulfilling the API contract specifications.

## PART 4 — DATABASES

**1. MongoDB (Primary Database)**
- **Stores:** User profiles, unstructured document metadata, flat compliance tasks, and audit logs.
- **Do NOT Store:** Highly interconnected relationship matrices or heavy blob files.
- **Why:** Its schema-less document nature perfectly matches the initially unstructured JSON outputs generated by the AI Engine.

**2. Neo4j (Graph Database)**
- **Stores:** The relationships between Organizations, Regulations, specific Clauses, extracted Obligations, and operational Workflows.
- **Do NOT Store:** Large text blocks, raw files, or volatile session data.
- **Why:** Regulatory compliance is highly relational. Graph queries easily traverse links like "Find all intermediary tasks affected by the amendment to Master Circular X".

**3. Redis (Cache & Message Queue)**
- **Stores:** Ephemeral JWT blocklists, API rate limiting counters, and Celery task queues.
- **Do NOT Store:** Persistent business data or audit logs.

**4. Qdrant (Vector Database)**
- **Stores:** Dense vector embeddings of regulatory clauses generated by the embedding model.
- **Do NOT Store:** Primary relational data or application state.
- **Interaction:** The backend queries Qdrant with a vectorized user query to retrieve similar legal clauses, which are then fed into the LLM as context (RAG).

## PART 5 — AI STACK

- **LangGraph**: Serves as the agentic orchestrator. Unlike simple sequential chains, LangGraph allows us to build stateful, cyclic workflows (e.g., Extract -> Validate -> If poor confidence, Re-Extract).
- **all-MiniLM-L6-v2**: A highly efficient sentence-transformer model used to locally generate vector embeddings for text chunks without incurring API costs or latency.
- **Configurable LLM**: The system allows swapping between OpenAI (GPT-4), Gemini, or Local Models (e.g., Llama 3 via Ollama) depending on deployment security and privacy constraints.

**How it works:**
1. Text is chunked and embedded via `MiniLM`.
2. Embeddings are stored in Qdrant.
3. LangGraph agents query Qdrant, build a context window, and instruct the LLM to extract structured Obligations.

---

## PART 6 — DOCUMENT PROCESSING

- **PyMuPDF**: The primary, lightning-fast parser used to extract raw text, tables, and metadata from standard, text-selectable SEBI PDFs.
- **Tesseract OCR (Optional)**: Invoked only as a fallback when PyMuPDF detects scanned images with no embedded text.

**Failure Handling:**
If both PyMuPDF and OCR fail to extract meaningful text, the Document API flags the file status as `FAILED_EXTRACTION` and alerts the user to upload a cleaner copy.

---

## PART 7 — BACKGROUND PROCESSING

- **Celery**: The standard distributed task queue for Python. Used to offload heavy operations (AI Extraction, Bulk Task Generation, Daily Reminder Emails) from the main FastAPI thread.
- **Redis Broker**: Acts as the intermediary holding the queue of tasks waiting to be processed by Celery workers.

**Retry & Failure Handling:**
Celery tasks must be idempotent. If an AI extraction fails due to a network timeout with an external LLM, Celery will use exponential backoff to retry the task up to 3 times before moving it to a Dead Letter Queue (DLQ).

## PART 8 — DEPLOYMENT STACK

- **Docker**: Every service (Frontend, FastAPI, Celery Worker, MongoDB, Neo4j, Qdrant, Redis) is containerized via a `docker-compose.yml` for unified local development and deployment.
- **Nginx**: Acts as the reverse proxy gateway. It terminates SSL/TLS, serves static frontend assets (in production), and routes `/api/*` traffic to the FastAPI containers.

## PART 12 — BEST PRACTICES

**Python/FastAPI Coding Standards:**
- Strictly use `async def` for any route that interacts with MongoDB, Redis, or external APIs. Use standard `def` only for purely CPU-bound synchronous tasks.
- Keep route controllers thin. Inject service classes to handle business logic.
- Use Pydantic models for *all* incoming/outgoing data validation.

**React Coding Standards:**
- Favor Functional Components and Hooks over Class Components.
- Use explicit TypeScript interfaces for all props and state variables.

**Security:**
- Never commit `.env` files.
- Validate all JWTs using FastAPI dependency injection (`Depends(get_current_user)`).
- Escape and sanitize all inputs before writing to Neo4j to prevent Cypher injection.

## PART 13 — IMPLEMENTATION GUIDELINES

1. **How Services Communicate:**
   The frontend communicates with FastAPI via REST. FastAPI communicates with Celery asynchronously via Redis. Background Celery workers communicate directly with MongoDB/Neo4j to update states.

2. **Database Access Organization:**
   Database access must be abstracted behind the Repository Pattern. For example, `MongoDocumentRepository` and `Neo4jObligationRepository`. Services should not write raw queries.

3. **Background Job Implementation:**
   Celery tasks should be placed in `worker/tasks.py`. They must remain stateless and idempotent. Any required database state must be loaded at the start of the task execution using the provided ID.

4. **AI Component Interaction:**
   LangGraph state variables should be strictly typed using `TypedDict`. The LLM interactions should be wrapped in robust error handling to account for rate limits and hallucinated JSON outputs.

5. **Vector Search Usage:**
   When searching Qdrant, always apply metadata filters (e.g., filtering by `regulation_id`) *before* applying the KNN vector search to drastically improve query performance and accuracy.
